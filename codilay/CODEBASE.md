## Development Utilities: Aggressive Clean Script
> File: `clean.sh`

Located at the project root (`clean.sh`), this bash script is a development utility designed to forcefully eradicate all build artifacts, dependency caches, and plugin registries. It is primarily used to resolve stubborn compilation or linking issues, particularly within the macOS/CocoaPods build pipeline.

### Core Operations
The script executes a 9-step "aggressive" cleaning process:
1. **Standard Clean**: Initiates a standard `flutter clean`.
2. **Artifact Removal**: Manually deletes `build/`, `macos/build/`, and `.dart_tool/` directories.
3. **Plugin Registry Wipe**: Removes `.flutter-plugins`, `.flutter-plugins-dependencies`, and the ephemeral macOS Flutter directory to force a full regeneration.
4. **CocoaPods Purge**: Deletes `macos/Pods`, `.symlinks`, and `Podfile.lock`.
5. **Deep Cache Clearing**: Navigates into the `macos` directory to run `pod cache clean --all` and `pod deintegrate`.
6. **Local Pub Cache Clearing**: Specifically removes cached versions of the `local_notifier` plugin from the global `~/.pub-cache`.
7. **Dependency Resolution**: Runs `flutter pub get` to regenerate freshly downloaded packages.
8. **Local Plugin Verification**: Greps the newly generated `.flutter-plugins` file to ensure a specific custom local plugin (`local_notifier=packages/local_notifier_custom`) is correctly recognized by the framework.
9. **Pod Reinstallation**: Executes `pod install --verbose` to rebuild the iOS/macOS native dependency tree.

### Usage & Context
This script highlights an architectural reliance on a custom or heavily modified local dependency (`packages/local_notifier_custom`). Native development setups (specifically Xcode and CocoaPods) often aggressively cache plugin locations, leading to "plugin not found" errors when modifying local Flutter packages. Running this script ensures a pristine slate before compiling via `flutter build macos` or directly through Xcode (`macos/Runner.xcworkspace`).

## Cross-Platform Foreground Window Plugin
> File: `lib/foreground_window_plugin.dart`

### `ForegroundWindowPlugin`
Located in `lib/foreground_window_plugin.dart`, this module serves as a cross-platform facade for retrieving real-time data about the currently active (foreground) window on desktop environments.

**Platform Implementations:**
- **Windows:** Delegates directly to a native FFI (Foreign Function Interface) implementation mapped in `foreground_window_plugin_windows.dart`.
- **macOS:** Utilizes Flutter's `MethodChannel` (`foreground_window_plugin`) to asynchronously invoke the `getForegroundWindow` method, interfacing directly with a Swift native plugin.
- **Other Platforms:** Throws an `UnsupportedError`.

**Data Retrieval Flow:**
Calling `getForegroundWindowInfo()` returns a `WindowInfo` object containing system-level details, including:
- `windowTitle`
- `processName` and `executableName`
- `programName`
- `processId` and `parentProcessId`
- `parentProcessName`

To simplify dependency management across the app, this file additionally re-exports the `WindowInfo` and `AppLaunchInfo` models from the Windows-specific implementation file.



→ See [Windows Foreground Tracker (FFI)](#windows-foreground-tracker-ffi) for the low-level Windows FFI implementation and the `WindowInfo` and `AppLaunchInfo` definitions.

## Windows Foreground Tracker (FFI)
> File: `lib/foreground_window_plugin_windows.dart`

### `ForegroundWindowPlugin` (Windows)
Located in `lib/foreground_window_plugin_windows.dart`, this file implements the Windows-specific mechanisms for tracking active applications. It uses `dart:ffi` to bind directly to standard Windows system APIs, which replaces the need for standard C++ method channels for window tracking.

#### Core Native Bindings
The plugin lazily loads and interfaces with several Windows DLLs:
- **`user32.dll`**: Resolves the active window (`GetForegroundWindow`, `GetWindowTextA`, `GetWindowThreadProcessId`).
- **`kernel32.dll`**: Handles process access and snapshots (`OpenProcess`, `CloseHandle`, `CreateToolhelp32Snapshot`, `Process32First/Next`).
- **`psapi.dll`**: Retrieves executable paths (`GetModuleFileNameExA`).
- **`version.dll`**: Extracts human-readable application names from PE file resources (`GetFileVersionInfoA`, `VerQueryValueA`).

#### Data Models
- **`WindowInfo`**: Encapsulates data about the active window, including `windowTitle`, `processName`, `executableName`, `programName`, and process IDs.
- **`AppLaunchInfo`**: Represents metadata about how an application was launched, supplying the data structure used by the Launch Detection mechanisms.

#### Execution & Heuristics
To prevent dropping frames or blocking the Flutter UI thread, the core native logic (`_getForegroundWindowInfoNative`) is executed in a background isolate using Flutter's `compute` function. 

The plugin includes intelligent parsing and fallbacks:
1. **Version Resource Parsing**: Tries to read `FileDescription` or `ProductName` from the executable's native metadata (handling multiple language codes) to get a clean app name (e.g., "Google Chrome" instead of "chrome.exe").
2. **Protected Processes**: If a process is elevated or protected (preventing `OpenProcess`), it falls back to a heuristic (`_heuristicProgramName`) that strips common browser/app suffixes from the window title (e.g., "My Document - Microsoft Word" → "Microsoft Word").

## Background App Tracker Controller
> File: `lib/sections/controller/application_controller.dart`

The `BackgroundAppTracker` (located in `lib/sections/controller/application_controller.dart`) is a core singleton service responsible for monitoring active applications, tracking screen time, and managing the state of background usage collection.

### Key Responsibilities
- **Screen State Management**: Integrates with the `desktop_screenstate` plugin to detect `sleep`, `screenOff`, and `locked` states. Tracking is only allowed when all three are false (`_screenStateAllowsTracking`). It implements a 500ms debounce on `unlocked` and `awaked` events to filter out spurious events emitted by macOS during the lock/wake cycle. When a pause event occurs in `precise` mode, it commits the current app's elapsed time before flipping the state flag to avoid losing data.
- **Window Focus Tracking**: Leverages the `WindowFocus` plugin to listen for focus changes (`_handleFocusChange`) and idle detection (`_handleUserActivityChange`). It can be configured via `SettingsManager` to monitor audio levels, game controllers, HID devices, and keyboard/mouse input to determine user presence. It sanitizes window titles by stripping null bytes, zero-width characters, and control characters, and handles edge cases for Windows system apps (e.g., `SearchHost`, `Application Frame Host`).
- **Tracking Modes**: 
  - **Polling Mode**: Runs a 1-minute periodic timer (`_pollingTimer`) that fetches the current active app. If the system is active and the user is not idle, it records a fixed 1-minute usage block.
  - **Precise Mode**: Event-driven tracking based on window focus changes. Uses a 60-second "universal heartbeat" (`_universalHeartbeat`) to periodically commit elapsed time for the current app. If the user is focusing the tracker app itself, it switches to a faster 5-second "self-tracking heartbeat".
- **Time Aggregation**: Calculates elapsed time using `DateTime.now().difference(_currentAppStartTime)`. Usage is recorded to the `AppDataStore` via `recordAppUsage`, pushing both aggregate time and exact `TimeRange` blocks. Tracking is skipped if the app's metadata indicates it should be hidden or ignored (`isTracking == false` or `isVisible == false`).
- **Permissions**: Verifies and requests necessary system permissions (e.g., Input Monitoring, Screen Recording on macOS) via `checkAllPermissions()` and `checkInputMonitoringPermission()`.
- **Categorization & Localization**: Loads localized app names and applies categorization rules (`_categorizeAppWithLocale`, `_categorizeAppEnglishOnly`) via the `CategoriesController`.

### Notable Methods
- `initializeTracking({String? locale})`: Bootstraps the tracking system, loads localization from `SettingsManager`, hydrates the `_metadataCache` from `AppDataStore`, initializes plugins, and starts the configured tracking mode.
- `setTrackingMode(TrackingMode mode)`: Transitions between Precise and Polling modes, stopping the current mode cleanly and updating settings.
- `_handleFocusChange(AppWindowDto window)`: Triggered when the user switches applications. Sanitizes the title, updates the internal start timer, and requests metadata if the app is new.
- `_handleUserActivityChange(bool isActive)`: Pauses precise tracking when the user becomes idle and resumes when active.
- `_applyScreenStateEvent(String event)`: Core state machine for system pause/resume events, responsible for committing time before system sleep/lock.

nd granular `TimeRange` blocks. Upon committing time, it checks the app's metadata (`isTracking` and `isVisible` flags) and, if valid, writes to the `AppDataStore`. It subsequently triggers the `NotificationController` to evaluate daily usage limits and dispatch notifications if necessary.

### Metadata & Categorization
The tracker automatically classifies newly detected apps to determine their productivity status:
- **Initialization & Caching**: The `_getOrCreateMetadata()` method checks `AppDataStore` for existing metadata or creates a new entry, caching results in `_metadataCache`. It explicitly ignores the tracker app itself (e.g., "Productive ScreenTime").
- **Auto-Categorization**: Unrecognized apps are categorized by matching window titles against categorized app lists defined in `AppCategories`.
- **Localization Support**: Leveraging `AppLocalizations`, it translates known app names (e.g., "System Preferences", "Calculator") to match localized category lists, ensuring robust categorization regardless of the host OS language.
- **Productivity Defaults**: Any app categorized as `Social Media`, `Entertainment`, `Gaming`, or `Uncategorized` defaults to non-productive (`isProductive = false`). All others default to productive.

### Settings & OS Integration
The tracker bridges user configuration (`SettingsManager`) and native OS plugins:
- Dynamically applies idle detection timeouts and hardware monitoring toggles (Audio, HID, Keyboard/Mouse, Game Controllers) directly to the `WindowFocus` plugin without requiring a restart.
- Wraps system-level permission checks (Screen Recording, Input Monitoring) and exposes functions to open OS settings panels.
- Uses `ForegroundWindowPlugin.getForegroundWindowInfo()` as a fallback or helper to manually fetch active window states.

### Lifecycle & Shutdown
When `stopTracking()` is called, the controller ensures zero data loss by:
1. Deregistering screen state listeners.
2. Saving the final elapsed time for the current active app.
3. Halting background timers and event streams.
4. Invoking `AppDataStore.forceCommitToHive()` to synchronously flush pending usage data to disk.
5. Disposing of native plugin resources.

  - `HIDE`: Hides the application window.
  - `EXIT`: Gracefully terminates the application.
- **Cross-Platform Window Management**: Depending on the operating system, it delegates window actions:
  - On macOS, it utilizes the `MacOSWindow` utility (→ See [Utilities: macOS Window Manager]) to interface with the native window manager.
  - On Windows and Linux, it relies on the `bitsdojo_window` package's `appWindow` API to show, restore, hide, or close the application.
- **Client Requests**: Secondary instances utilize the `requestShow`, `requestHide`, and `requestExit` methods. These establish a brief `Socket` connection to the primary instance, transmit the corresponding command, and immediately close the connection.

### Usage Context
This utility is crucial for preventing multiple instances of the app from running simultaneously and competing for access to the local Hive database. During application launch, if binding the IPC server fails (indicating another instance is already running), the secondary instance can send a `SHOW` request to the primary instance and exit, creating a seamless user experience.

## Utilities: Update Service
> File: `lib/utils/update_service.dart`

Located in `lib/utils/update_service.dart`, the `UpdateService` is a singleton `ChangeNotifier` responsible for checking, caching, and exposing application update availability.

### Core Functionality
- **Platform-Aware Checking**: Queries a central endpoint (`https://api.scolect.com/update`) appending the current OS platform (`macos` or `windows`) to ensure only relevant updates are fetched.
- **Periodic Polling**: Automatically polls for updates every 6 hours using a `Timer.periodic`, provided it is initialized correctly on startup.
- **Caching Mechanism**: Uses `SharedPreferences` to cache the last checked timestamp and any available `UpdateInfo`. This allows the application to instantly display an update banner on a fresh launch without waiting for a network request to complete.
- **Version Comparison**: Parses semantic version strings (e.g., `1.2.3`) to reliably determine if a remote tag is newer than the locally running instance.

### State Management
The service manages an `UpdateStatus` (`idle`, `checking`, `upToDate`, `updateAvailable`, `error`) and notifies listeners (primarily the About Settings section) whenever the status or the underlying `UpdateInfo` changes.

### Initialization
The service is expected to be initialized exactly once at application startup (typically from `main.dart`) by passing the current running version string: `UpdateService().initialize(currentVersion)`. This initialization kicks off the background timer and immediately restores any cached update from disk.
